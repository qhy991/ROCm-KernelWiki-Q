# [ROCm][Perf] MXFP8 dense-linear + grouped-MoE GEMM optimizations for MiniMax-M3

Open PR #46117 in [vllm-project/vllm](https://github.com/vllm-project/vllm/pull/46117). This page records the proposal as an upstream work item, not a released vLLM feature.

**Author:** amd-ethany  
**Opened:** 2026-06-19  
**Status verified:** open on 2026-06-19

## What Changes

The PR targets the native MXFP8 path for MiniMax-M3 decode on AMD MI355X / CDNA4 (`gfx950`). It changes two serving hot paths:

- Dense MXFP8 linear in `vllm/model_executor/kernels/linear/mxfp8/rocm_native.py`.
- Grouped MoE MXFP8 dispatch and kernel code in `vllm/model_executor/layers/fused_moe/experts/mxfp8_native_moe.py`.

The motivating shape is decode: `M` is often tiny while projection `K` remains long. The PR proposes a thin-M/deep-K tile for dense linear and a more specialized grouped-MoE path instead of relying on a square-ish tile that wastes rows in single-token or small-batch decode.

## Kernel Relevance

For ROCm serving kernels, this is useful evidence that MiniMax-style MXFP8 decode needs separate treatment from prefill or training GEMM:

- Small `M` changes the right tile shape and occupancy balance.
- Grouped-MoE kernels need routing/top-k metadata to be visible at the GEMM boundary.
- The PR explicitly preserves the math contract around `tl.dot_scaled` and fp32 accumulation, so the change is a scheduling/layout optimization rather than a numerics change.
