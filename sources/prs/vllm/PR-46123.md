# [ROCm][Perf] Optional FlyDSL BF16 MoE for the MXFP8-emulation path on MiniMax-M3

Open PR #46123 in [vllm-project/vllm](https://github.com/vllm-project/vllm/pull/46123). It is opt-in and unmerged as of capture time.

**Author:** yueliu14  
**Opened:** 2026-06-19  
**Status verified:** open on 2026-06-19

## What Changes

This PR adds an optional FlyDSL-backed MoE route for MiniMax-M3 when vLLM is using the MXFP8 emulation path on MI300 / `gfx942`:

- New `flydsl_emulation_moe.py` driver for AITER vendored FlyDSL two-stage MoE GEMM kernels.
- Environment gate `VLLM_MINIMAX_M3_FLYDSL_MOE=1`; the default behavior remains unchanged.
- Runtime guards for BF16 weights, no expert parallelism, `topk=4`, and no router-weight-on-input.
- Fallback to the Triton path on unsupported shape, architecture, or runtime failure.

## Kernel Relevance

The PR is valuable for the ROCm serving-kernel wiki because it separates two common MiniMax-M3 cases:

- Native MXFP8 on CDNA4 can use scaled-MFMA-oriented kernels.
- MXFP8 emulation on CDNA3 may still benefit from BF16 two-stage MoE kernels when the quantized path is emulated by the framework.

It also documents a conservative integration style: keep the new MoE backend opt-in and heavily guarded until enough production coverage exists.

## Wiki Use

- `kernel-moe-grouped-gemm-cdna4`: contrast CDNA3 emulation MoE with CDNA4 native MXFP8 MoE.
- `kernel-rmsnorm-rocm`: related only when norm/quant producers feed the emulation path; do not cite it as an RMSNorm implementation.
