# [AMD][OCP MX][CI] Fix ROCm OCP MX emulation dispatch

Open PR #46142 in [vllm-project/vllm](https://github.com/vllm-project/vllm/pull/46142). It is open and unmerged as of capture time.

**Author:** fxmarty-amd  
**Opened:** 2026-06-19  
**Status verified:** open on 2026-06-19

## What Changes

This PR is partly CI/test cleanup and partly a real MoE quantization-path fix. The kernel-relevant pieces are:

- Forces OCP MX tests on MI250, MI300, and MI325 to exercise the MoE emulation path instead of `UNFUSED_TRITON`.
- Improves W-MXFP4-A-FP8 emulation so the AMD GPT-OSS W-MXFP4/A-FP8/KV-FP8 test meaningfully runs on MI300.
- Removes a duplicate `moe_kernel_quantize_input` call in `ocp_mx_emulation_moe.py`; the parent Triton MoE path already performs the input quantization.
- Updates MXFP4 oracle/util handling and Quark MoE integration around the emulated quantized path.

## Kernel Relevance

Treat this as an emulation-path correctness and dispatch source, not as a new kernel implementation. The PR is useful because it documents a failure mode for ROCm quantized MoE tests: a test can pass through an unfused Triton backend that does not perform required input quantization and therefore fails to validate the intended OCP MX emulation behavior.

For wiki use, cite it when discussing guarded MXFP4/FP8 emulation on older ROCm targets and the need to keep test dispatch aligned with the quantized MoE kernel contract.
