# [ROCm][Perf] MiniMax-M3 MXFP8 gemm/group gemm dispatch AITER

Open PR #46063 in [vllm-project/vllm](https://github.com/vllm-project/vllm/pull/46063). This is an unmerged dispatch proposal, so it should be cited as work-in-progress rather than shipped behavior.

**Author:** JohnQinAMD  
**Opened:** 2026-06-18  
**Status verified:** open on 2026-06-19

## What Changes

The PR adds runtime dispatch from vLLM's MiniMax-M3 native MXFP8 decode path into AITER small-M HIP kernels on `gfx950`:

- Dense linear tries AITER `mxfp8_gemv` after activation/shape guards.
- MoE grouped GEMM is renamed internally to separate the Triton implementation from a new dispatcher.
- The dispatcher tries AITER `grouped_gemm_mxfp8` when fused-MoE support is enabled and the request is compatible.
- Triton remains the fallback for unsupported architectures, shapes, or expert-map cases.

## Kernel Relevance

This PR is a clean example of serving-framework dispatch rather than pure kernel authoring. It captures the boundary between vLLM's Python routing and AITER's HIP kernel library:

- Keep a stable fallback path while enabling vendor/library kernels for narrow decode regimes.
- Gate AITER use on architecture and MoE metadata so unsupported cases remain correct.
- Thread `topk_ids` to the kernel boundary, which is required for grouped-MoE small-M scheduling.
