# Fix CK FP8 grouped GEMM dtype gating for columnwise operands

Merged PR #594 in [ROCm/TransformerEngine](https://github.com/ROCm/TransformerEngine/pull/594).

**Merged:** 2026-06-10  
**Base branch:** `dev`

## Status

GitHub reports this PR as closed and merged on 2026-06-10. It changed 4 files with about 185 additions and 38 deletions.

## What Changed

- Fixes FP8 grouped-GEMM dtype gating to use the effective operand dtype after canonicalization.
- Avoids incorrect rejection of some WGRAD columnwise FP8 operands by the CK grouped-GEMM backend.
- Adds FP8 padding support when required by kernel constraints.
- Adds shape specialization for fully aligned operands.
- PR description reports fallback warnings dropping from 7,680 to 960 in an E2E Qwen Megatron run; treat this as source-reported run evidence.

## Kernel Relevance

This is a dispatch/correctness fix for CK FP8 grouped GEMM. The key lesson is that tensor storage layout and canonicalized operand dtype must agree before deciding whether CK can handle a grouped GEMM; otherwise TransformerEngine may fall back to slower multi-stream hipBLASLt paths.
