# [AMD] Optimize o_proj gemm and attn output rope performance

Open PR #28722 in [sgl-project/sglang](https://github.com/sgl-project/sglang/pull/28722). It is still open as of this capture.

**Author:** kkHuang-amd  
**Opened:** 2026-06-19  
**Status verified:** open on 2026-06-19

## What Changes

The PR targets DeepSeek-V4 projection and attention-output glue on ROCm:

- For selected MLA/output/q/kv projection FP8 GEMM shapes, it prefers AITER's CK blockscale path over the existing Triton `w8a8` blockscale GEMM choice on `gfx950` with HIP 7.2 or newer.
- It adds a specialized attention-output inverse RoPE path because the CUDA fused path does not directly translate to HIP, where the fallback can become a per-token strided Triton path.

## Kernel Relevance

This PR is a good example of framework-level kernel selection:

- A hard-coded tuned-shape list can become stale when a library backend gets faster for the same shapes.
- RoPE after attention is not just elementwise glue if fallback dispatch creates many tiny programs.
- Serving performance depends on both large projection GEMMs and the launch/layout cost around them.
