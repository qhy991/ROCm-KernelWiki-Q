## What Changes

The PR addresses DeepSeek-V4 MXFP8 routed MoE on the `flashinfer_trtllm_routed` path after RL weight updates. The reported failure is not the steady-state kernel; it appears after weight reload, when cached GPU-resident row-permutation tensors used to align MXFP8 MoE weights/scales can be clobbered by reused weight memory.

The patch touches the MoE layer and runner plumbing so the layout-conversion cache is safe across weight updates.

## Kernel Relevance

This PR is useful because serving kernels often hide layout transforms behind Python dispatch. For routed MoE, the row permutation is part of the kernel contract:

- Weight/scales layout alignment must remain valid after online updates.
- GPU-side helper tensors are not harmless constants if the allocator reuses their memory.
- Correctness testing should cover reload/update paths, not just first-load inference.
