# Autotuning Artifact Management in Triton ROCm

## Overview

PR #663 in the Triton ROCm backend repository introduces a critical structural improvement to the autotuning infrastructure, specifically targeting the GEMM tuning pipeline (`tune_gemm`). The modification changes the output directory for autotuner artifacts from the root `tune_gemm` directory to a dedicated `tune_gemm/output` subdirectory.

## Architectural Intent and System Design

### 1. Separation of Concerns in the Autotuner
In the Triton compiler stack, the autotuner plays an essential role in finding the optimal kernel configuration (e.g., `BLOCK_SIZE_M`, `BLOCK_SIZE_N`, `BLOCK_SIZE_K`, `num_stages`, `num_warps`) for CDNA architectures (such as MI250X, MI300X). As the tuner sweeps through the search space, it generates numerous artifacts, including compiled binaries, PTX/ISA assembly dumps, and performance benchmark logs.

By isolating these outputs in `tune_gemm/output`, the PR enforces a clear separation between:
- **Executable tuning scripts** (`tune_gemm/`)
- **Generated autotuning artifacts** (`tune_gemm/output/`)

### 2. File I/O and Concurrency Improvements
When running extensive hyperparameter sweeps for large LLM workloads, the I/O generated by the Triton autotuner can become a bottleneck. Consolidating outputs to a single structured directory:
- Reduces directory parsing overhead during cache lookups.
- Prevents workspace pollution which can interfere with source control systems and CI/CD pipelines.
- Makes it trivial to purge, backup, or analyze a specific batch of tuning results by manipulating the `output/` directory directly.

## Performance and Memory Bound Context

While this PR is a host-side infrastructural change rather than device-side kernel code, it has upstream implications for kernel optimization loops. 

### Autotuner Search Space
GEMM performance on ROCm (using MFMA instructions) is extremely sensitive to memory bounds and register allocation. The autotuner evaluates kernels that span the spectrum of:
- **Memory-bound configurations:** Large block sizes that maximize LDS (Local Data Share) usage but may cause register spilling or drop occupancy.
- **Compute-bound configurations:** Configurations that maintain high occupancy and leverage Matrix Fused Multiply-Add (MFMA) instructions efficiently but may underutilize the HBM bandwidth.

By structuring the output cleanly, developers and CI systems can more easily parse the generated JSON logs or benchmark reports to build Pareto fronts and select the optimal configuration for a given matrix dimension and CDNA architecture.

## CI/CD and Automation Integration

For automated nightly tuning jobs targeting CDNA3 (MI300X) or future CDNA4 hardware, structured artifact directories enable:
1. **Automated Log Harvesting:** Tools can simply collect `tune_gemm/output/*.json` to track performance regressions over time.
2. **Deterministic Cleanup:** `rm -rf tune_gemm/output/*` ensures a clean state without risking deletion of tuning source files.
3. **Parallel Tuning Workspaces:** Future improvements can easily map `tune_gemm/output_$JOB_ID` to support concurrent tuning jobs on multi-GPU systems.

## Conclusion

Though superficially a path update, this PR represents a maturation of the Triton-ROCm autotuning pipeline. It aligns the tuning infrastructure with best practices for system design, enabling more robust, scalable, and automated kernel optimization workflows for AMD GPUs.
